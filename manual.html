U1快速開始（3 步驟上手）

這個工具能做什麼？

  • 讀取 IC 設計 .gds 檔案 → 自動辨識 Bump / Amark / Chip Size 等 Layer 用途
  • 分類 PAD：INPUT / OUTPUT / DUMMY，並自動標注 Amark 對齊標記
  • 產生 PAD Excel 報告（含尺寸總表 + 5 區簡圖 + 間距代號 S1, S2…）
  • 跑 Bump DRC 規則檢查（TM R1~R5、BOE B1~B9）→ HTML / PDF / CSV 報告
  • 渲染 GDS 為彩色 PNG 預覽圖

啟動步驟

  1. 雙擊專案根目錄的 START_SERVER.bat
  2. 等待瀏覽器自動打開 http://127.0.0.1:5000（10–30 秒）
  3. 上傳 .gds 檔案 → 點「掃描 GDS」→ 依需求點功能鈕

U4Layer 對應規則

優先級總覽

系統按以下順序判定 Layer 用途，前者命中則跳過後者：

  1. (Layer, DType) 精確強制（如 190:9 → chip_size）
  2. Layer 224/225 DType 0 特例（含 #POLY → bump；含 #TEXT → bump_text_number）
  3. Layer 強制覆蓋（如 224、225 → bump）
  4. 幾何自動推測（polygon 數量、面積、長寬比、頂點數）
  5. 使用者手動覆蓋（表格「修改」欄）

預設 Layer 對應

用途LayerDType說明
chip_size1909晶片邊界矩形（優先級最高）
amark205224Alignment Mark 外框
bump224, 2250Bump/PAD 幾何（#POLY 強制）
bump_text_number224, 2250Bump 序號（#TEXT 強制）
bump_text_name211224信號名稱（VDD、RX[0]…）

幾何自動推測（polygon）

條件推測用途
polygon 數 ≤ 4，平均面積 > 100,000 µm²chip_size
polygon 數 ≤ 8，最大頂點 ≥ 8（十字形），W/H ≤ 150 µmamark
polygon 數 ≤ 20，最小頂點 = 3（三角形），W/H ≤ 150 µmamark
polygon 數 > 4，平均面積 < 5,000 µm²，長寬比 0.5–2.0bump
其他unknown

文字內容推測（text）

特徵推測
amark layer（205）的附屬文字amark_text
含「NO」（如 NO3501）bump_text_number
含英文字母（RX[9]、VDDD）bump_text_name
純數字bump_text_number
Layer 224 與 225 的 DType 0 強制規則不受 mapping JSON 影響 — 若實際設計有特殊用途，需直接修改 gds_server.py 中的 LAYER_OVERRIDES。

U5PAD 分類邏輯

三大分類

類型判定條件
INPUT密集單排（同 Y 行 > 10 個，±0.5 µm 容差）+ 晶片下半部，固定 1 排
OUTPUT密集單排 + 在 INPUT 上緣以上，可為 1 排或多排
DUMMY其他：邊緣稀疏 / 角落稀疏 / IC 中間稀疏的 PAD

距離計算原則

  • BUMP to IC edge：PAD 邊到 chip 邊的最小值（real，非 avg）
  • OUTPUT to INPUT 間距：OUTPUT 最下點 − INPUT 最上點（只取靠近 chip 中心的 OUTPUT）
  • 所有距離計算一律使用 real（實際最小值），嚴禁 avg
嚴禁使用平均值計算邊界距離。若 PAD 排列不齊，avg 會掩蓋實際的最小違規距離， 導致 DRC 漏檢。任何看似「合理的平均化」都會破壞此規則。

U6Excel 輸出格式

Sheet 1 — PAD 清單

符號說明計算
A1~A3INPUT PAD 尺寸W×H 統計，按數量排序
A4INPUT 到 IC 下方邊min(INPUT 底部 → chip 下邊)
A5INPUT 到 IC 側邊min(INPUT 左右邊 → chip 側邊)
B1~B2OUTPUT PAD 尺寸W×H 統計
B3OUTPUT 到 IC 上方邊min(OUTPUT 頂部 → chip 上邊)
B4OUTPUT 到 IC 側邊min
B5OUTPUT 到 INPUT 間距最下 OUTPUT − 最上 INPUT
C1~C2DUMMY PAD 尺寸W×H 統計
C3DUMMY 到 IC 上方邊min
C4DUMMY 到 IC 側邊min

Sheet 2 — 簡圖及標注

  • 5 個區域：#1 左邊角、#2 左偏中、#3 中心（固定 ±1750µm）、#4 右偏中、#5 右邊角
  • 間距以 S1, S2, S3… 代號標注，相同間距（±0.001 µm）共用同一代號
  • 區域寬度 window_um 預設 3500 µm，可在左側面板調整 1000–10000
調大 window_um → 更廣的上下文；調小 → 更聚焦轉變點（如 INPUT/OUTPUT 邊界）。 A2 Center 區（#3）不受此設定影響，永遠是 ±1750µm。

U7Bump Rule 檢查（DRC）

規則套件

套件規則範圍主要檢查
TM（Timing Margin）R1 ~ R5有效面積 / ILB pitch / ILB-OLB gap / Edge 距離 / B/(A+C) ratio
BOE（Body-on-Edge）B1 ~ B9IC size / edge bump 距離 / lead width / lead space / pitch / area

TM 規則摘要

編號規則閾值
R1Bump 有效面積eff_area ≥ 500 µm²
R2ILB pitch（主排平均）依設計規範
R3ILB → OLB 垂直 gap依 OLB 排數動態決定
R4Bump 到晶片邊距離≥ 40 µm（左右上下四邊）
R5B / (A+C) 比值< 5

BOE 規則摘要

編號規則閾值
B2邊緣 bump 到 IC 左/右邊≤ 200 µm
B3Lead width（短邊）≥ 12 µm
B4Lead space（pitch − avg_width）≥ 12 µm
B5Lead pitch≥ 15 µm
B6Lead area（c × a）≥ 950 µm²（當 c ≥ 65 µm）

輸出檔案

  • {name}_report.html — 完整 DRC 報告（自動開啟瀏覽器）
  • {name}_report.pdf — A4 版面（需安裝 Chrome）
  • {name}_tm_violations.csv / {name}_boe_violations.csv — 機器可讀的違規清單
每條規則的閾值寫死在 check_bump_rule2.pl 中。若 PDK 規範變更， 需直接修改 Perl 原始碼（見 D6 · Perl Rule Checker）。

U8操作技巧

典型工作流程

  1. 啟動 → 上傳 GDS → 掃描 GDS
  2. 檢查右側 Layer Mapping 表 → 必要時修改下拉 → 儲存 Mapping JSON
  3. Bump / Amark 分析 → 確認 INPUT/OUTPUT/DUMMY 數量合理
  4. 輸出 PAD Excel → 開啟資料夾檢視 .xlsx
  5. （可選）Bump Rule 檢查 → 看 HTML 報告中的紅色違規

常用快捷

  • 拖曳 GDS 到上傳框 → 等同點 Browse
  • 同一 GDS 重跑 Excel：直接點按鈕，輸出檔自動加 _v2 後綴
  • F12 → Console：查看分類與計算的詳細 log（標籤 [SCALE CHECK] [SPATIAL CLASS] [PITCH CALC] [EDGE_DISTS] [B4 WRITE]）

節省時間的建議

  • 同產品系列的 GDS 可重複利用 mapping JSON：先做一份「乾淨」的，後續加載即可
  • 跑大型 GDS（> 10000 bumps）前，先用「GDS 轉圖片」確認 chip 形狀正常
  • Excel 顏色：藍色 = INPUT、橘色 = OUTPUT、灰色 = DUMMY、紅色 = Amark

D3資料 Pipeline 全鏈

從 .gds 到 .xlsx

1. 上傳：POST /api/upload_gds (multipart) → 存到 input/
2. 掃描：POST /api/scan {path} → gdstk.read_gds → 提取 layers + polygons + texts
                              → _guess() 推測各 Layer 用途
                              → 回傳 layer 表 + 基本資訊
3. （可選）使用者調整 → POST /api/save_mapping → output/{name}.layer_mapping.json
4. 分析：POST /api/run_elements {path, mapping}
        → 套用 mapping → 取 chip_size 邊界 → 分類所有 bump：
            INPUT (Y < chip.center_y, 同行 > 10)
            OUTPUT (Y ≥ INPUT 上緣, 同行 > 10)
            DUMMY (其他)
        → 回傳 counts / amark 位置
5. Excel：POST /api/run_xlsx {path, mapping, window_um}
        → _build_xlsx() 計算 A1~C4 各距離
        → openpyxl 寫 Sheet 1 + Sheet 2（5 區簡圖 + S1~Sn 代號）
        → output/{name}.xlsx

從 .gds 到 DRC 報告

POST /api/run_bump_check {path}
  → subprocess: perl/perl/bin/perl.exe check_bump_rule2.pl --gds {path} --out output/
  → Perl 純手寫 parser（不依賴 gdstk）：
      decode_real8 / GDS record loop / 收集 boundary + text + sref
  → 算 TM R1~R5、BOE B1~B9
  → 輸出：
      output/{stem}_tm_violations.csv
      output/{stem}_boe_violations.csv
      output/{stem}_report.html
  → Python：若有 Chrome → headless print to {stem}_report.pdf
Perl 端用原始 GDS stem作為輸出前綴。Python 端有「重命名 → 原檔名」邏輯， 若改 Python 上傳目錄/暫存命名，需同步檢查 basename($gds_file, '.gds') 是否仍對齊。

D5Layer 推測演算法

函式入口：_guess(lyr, dt, polys, texts, scale, ctx_polys=None)

位於 gds_server.py:57。回傳字串 purpose（PURPOSES 中之一）。判定順序：

  1. (Layer, DType) 精確覆蓋（LAYER_DTYPE_OVERRIDES）—— 如 (190, 9) → chip_size
  2. Layer 224/225 DType 0 特例—— polygon 直接判為 bump；text 判為 bump_text_number
  3. Layer 強制（LAYER_OVERRIDES）—— 如 224, 225 → bump
  4. 幾何判斷（polygon 群統計）：
    • cnt ≤ 4 && avg_a > 1e5 → chip_size
    • cnt ≤ 4 && max_n == 4 && avg_a < 150 && max_w < 15 → origin_mark
    • cnt ≤ 8 && max_n ≥ 8 && max_w ≤ 150 → amark（十字形）
    • cnt ≤ 20 && min_n == 3 && max_w ≤ 150 → amark（三角）
    • cnt > 4 && avg_a < 5000 && 0.5 ≤ avg_ar ≤ 2.0 → bump
    • 其他 → unknown
  5. 文字判斷（無 polygon、只有 text）：
    • amark layer 附屬 → amark_text
    • 含 "NO" 或 純數字 → bump_text_number
    • 含字母 → bump_text_name

關鍵常數

# gds_server.py:41
LAYER_OVERRIDES = {
    204: "bump",
    224: "bump",   # 大量 polygon = Bump/PAD
    225: "bump",
}

# gds_server.py:48
LAYER_DTYPE_OVERRIDES = {
    (190, 9): "chip_size",
}

# gds_server.py:53
LAYER_TEXT_OVERRIDES = {
    225: None,   # 交由內容偵測
}
若導入新 PDK，建議優先擴充 LAYER_DTYPE_OVERRIDES（精確、不誤傷其他設計）， 而不是動 LAYER_OVERRIDES 或幾何閾值。

D6Perl Rule Checker

為何用 Perl？

  • 純手寫 GDS parser → 不依賴 gdstk 的版本差異
  • 歷史遺留：原始規則由 PDK team 用 Perl 維護，邏輯逐字翻譯
  • 啟動快、記憶體小，適合 CSV 風格的批次違規輸出

呼叫方式

perl/perl/bin/perl.exe check_bump_rule2.pl \
  --gds  <path/to/design.gds> \
  --out  <output/dir>

# 輸出（自動使用 gds 檔名 stem）：
# {stem}_tm_violations.csv
# {stem}_boe_violations.csv
# {stem}_report.html

核心流程

1. open GDS, byte-by-byte 解析 record（HEADER/BGNSTR/BOUNDARY/PATH/SREF/TEXT/...）
2. decode_real8() → 還原 8-byte GDS real
3. 累積每個 STRUCT 的 shapes 與 texts
4. 找 top struct（優先 'PAD'）
5. 依 Layer 分組 bumps，計算：
   - ILB pitch / OLB pitch
   - 主排 ILB y_max（用於 R5 ratio 範圍）
   - 邊距 A_y, C_y
   - 每個 bump 的 per-bump max（B_bump_max）
6. TM 區段：R1 → R2 → R3 → R4 → R5
   BOE 區段：B1 → B2 → ... → B9
7. 寫入 CSV：layer, bump_idx, rule, severity, ...
8. 組 HTML：含目錄、各規則表格、違規高亮

規則閾值（節錄，硬編碼於 .pl）

規則位置（行）閾值
R1 EffArea~487500 µm²
R4 Edge522–52540 µm（4 邊獨立）
R5 ratio~727B/(A+C) < 5
B3 lead width~628≥ 12 µm
B4 lead space~638≥ 12 µm
B5 lead pitch~656≥ 15 µm
B6 lead area~670≥ 950 µm² (c ≥ 65)
不要動 decode_real8() — 它是 IEEE 8-byte real 的逆變換， 錯一個 bit 整個 layout 座標都會亂掉。 若 GDS scale 不對（scale_um），先檢查 UNITS record 而非改解碼。

啟動腳本邏輯（START_SERVER.bat）

  1. 檢查系統 Python 安裝
  2. pip 檢查上述套件，缺失則 install
  3. 啟動 python gds_server.py
  4. 3 秒後 start http://127.0.0.1:5000

D8詞彙表

詞意義
Bump晶片表面用於封裝接合的金屬凸點，本工具中等同 PAD
PADI/O 接點，分 INPUT / OUTPUT / DUMMY 三類
AmarkAlignment Mark，光刻對齊用，通常十字 / 三角形狀
ILBInner Lead Bond — 內側 bump（靠近 chip 中心）
OLBOuter Lead Bond — 外側 bump（靠近 chip 邊緣）
TMTiming Margin 規則套件（R1~R5）
BOEBody-on-Edge 規則套件（B1~B9）
DRCDesign Rule Check — 設計規則檢查
GDSII業界標準 IC layout 二進位格式
DTypeGDS 中 Layer 的 datatype 子分類
scale_umGDS UNITS record 轉 µm 的縮放因子（典型 0.001）
S1, S2…Excel 簡圖中代表「等距」的代號（±0.001 µm 容差）
